Can one transaction in a typical NoSQL columnar database running on a cluster of servers span changes in many records? Explain why transactional processing is implemented the way it is in such databases.

### Answer

In a typical wide-column NoSQL store (e.g. Apache Cassandra, HBase) a transaction usually **cannot freely span many records (rows) across the cluster** with full ACID guarantees. Atomicity is normally provided only at the level of a **single row (or single partition/key)**.

The reason is the distributed, partitioned, horizontally-scaled design. Rows are sharded across many nodes by their key, so a multi-row transaction would require coordinating locks/commits across several machines (e.g. distributed two-phase commit), which is expensive and harms availability and latency. Following the CAP trade-off, these systems prioritise **availability, partition tolerance and scalability** over strong multi-record consistency. Restricting atomic operations to a single row/partition keeps writes fast and the cluster highly available, at the cost of cross-record transactional guarantees (which applications are expected to design around).
